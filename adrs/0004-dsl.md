## Context

ADR-0001 ratified: single choreography `.feature` per logical scenario, actor-addressed steps, screenplay total order on a DAG scheduler, detached-task and signal escape hatches, custom runner over the official Gherkin parser stack, 100% standard Gherkin. This ADR fixes the concrete step-registration API, the compiler, the built-in vocabulary, and the static analysis — precisely enough to implement `@kraken/gherkin`.

## Decisions

### D1 — Step registration API (context-first, expression-matched, metadata-carrying)

```ts
import { createStepRegistry } from '@kraken/gherkin';
export const { Given, When, Then, defineParameterType, registry } = createStepRegistry();

When('{actor} sends the message {string}', async ({ actor }, message) => {
  await actor.session.typeText({ by: 'testId', value: 'composer' }, message);
  await actor.session.tap({ by: 'testId', value: 'send' });
});

Then('{actor} sees the message {string} within {duration}',
  { polls: true },
  async ({ actor }, text, timeout) => { /* polling assertion helper */ });

When('{actor} announces {string}',
  { publishes: ['$1'] },              // metadata for the dry-run reachability analysis
  async ({ actor }, name) => { await actor.signals.publish(name); });
```

- Handlers are **context-first functions** (no `this`; arrow-friendly). The context is core's `StepRunContext = { actor, world, tasks, abort, actors }` — signals and log ride on the actor (`actor.signals`, `actor.log`), exactly as the examples below show. `actor` is resolved from the `{actor}` capture (by convention the first capture; steps without an `{actor}` capture are rejected at registration — every Kraken step is addressed to exactly one actor).
- Matching via `@cucumber/cucumber-expressions` with custom parameter types: `{actor}` (matches a bare word or quoted name; validated against the closed actor set at compile time with did-you-mean) and `{duration}` (`10s`, `500ms`, `2m` → milliseconds).
- Registration options: `publishes: string[]` (signal names, `'$1'` referencing a capture) for reachability analysis; `detached: true` + `handle:` capture for background tasks; `polls: true` documents a polling assertion (Phase 1 records it; lint enforcement later).
- The exported names are literally `Given`/`When`/`Then`/`defineParameterType` — shaped for discovery by the official Cucumber VS Code extension (per the language-service spike; its findings and the exact `.vscode` settings are appended to this ADR).

### D2 — Compiler: pickles → `ScenarioPlan`

Parse with `@cucumber/gherkin` (AST → pickles; Background steps and Examples expansion come free), filter with `@cucumber/tag-expressions`, match each pickle step against the registry, and emit the core's `ScenarioPlan`: a **chain** of `step` nodes in text order (screenplay total order), with `detach` steps becoming `detach` nodes and join steps becoming `join` nodes. Actor extraction is static (from `{actor}` captures), so the orchestrator boots exactly the sessions the scenario needs, in parallel, before the first step.

### D3 — Dry-run analyzer (runs implicitly before every execution)

`compileFeatures(sources, registry, actors)` returns `{ plans, diagnostics }` and performs, with **no sessions booted**: unmatched-step detection (with the closest expression suggested); unknown-actor detection (did-you-mean over the closed set); detached-task leak detection (unjoined handle = error) and join-without-detach; **signal-wait reachability** (a wait step whose signal name has no `publishes` producer earlier in the chain nor in any detached task started earlier = guaranteed deadlock under total order = compile error); duplicate/ambiguous expression detection. `kraken run` runs this analysis first and refuses to start on errors; `--dry-run` stops after it.

### D4 — Built-in vocabulary (deliberately minimal)

`@kraken/gherkin` ships only choreography/system steps; app-domain steps belong to the user's project (ADR-0001 §5.9):

- `Then {actor} waits for the signal {string} within {duration}` (+ `... from {actor}` variant) — the v2-continuity step.
- `When {actor} starts {string} in the background as {string}` is **not** shipped generically (a background task needs an app-domain action); instead the `detached: true` registration option lets user steps become detachable, and the generic join step ships: `Then {actor}'s background task {string} completes within {duration}`.
- No generic UI steps (`taps "Send"`) in v1 — the step-library-philosophy question from ADR-0001 §9.2 resolves as: **core vocabulary = choreography only**; a clearly-labeled teaching pack may come later as a separate opt-in package, never as default vocabulary.

### D5 — Language policy (ADR-0001 §9.2, resolved)

**English-only step expressions** in Kraken's own packages, consistent with C12. Spanish-speaking teams get: Gherkin's `# language: es` dialect for keywords (free, standard), and the ability to register their own Spanish expressions for *their* app-domain steps (user projects are outside C12). Dual EN/ES expressions for Kraken built-ins would need a human-ratified C12 relaxation (per ADR-0001 §9.2) — not pursued; can be revisited with classroom evidence.

### D6 — Timeouts

Signal-wait and join steps carry their duration **in the step text** (self-documenting, per ADR-0003 D4's no-library-default rule). Polling assertion helpers take their timeout from an explicit argument or the project-level `defaults.assertionTimeoutMs` in `kraken.config.ts` (the one sanctioned config default — assertions are app-latency-bound, not choreography-bound).

### D7 — What Phase 1 does NOT include (recorded to prevent scope drift)

`requires:` capability constraints on steps (needs real capability interfaces — Phase 2, with drivers); `@cucumber/messages` envelope emission (Phase 4, ADR-0001 §5.12); matrix permutation runs (Phase 3); concurrent per-actor lanes (deferred behind the corpus review — appendix below decides).

## Consequences

- Every-step-has-an-actor is a hard rule; purely narrative steps (`Given the backend is seeded`) must name a responsible actor or use a future system-actor convention — revisit if it chafes in real suites.
- `publishes` metadata is honor-system (a handler could publish undeclared names at runtime); the analyzer catches the deadlock class that matters (waits with no declared producer), and runtime near-miss diagnostics (ADR-0003 D4) cover the rest.
- The registry is intentionally instance-based (`createStepRegistry()`) rather than global — no import-order magic, trivially testable, multiple isolated registries in one process (needed by the CTK and by user projects embedding Kraken).

## Appendix A — v2 scenario-corpus review (D6 evidence, mandated by ADR-0001 §5.9/§7)

**Status:** Complete (2026-07-03). **Reviewed by:** Claude (Fable 5), Phase 1.
**Question under test (ADR-0001 §5.9):** was v1/v2's concurrency an artifact of the N-process architecture, or a property users needed? **Decision threshold (§9.2):** if >~10% of real scenarios genuinely need sustained simultaneity, pull concurrent lanes into v3.0.

### A.1 Methodology

1. **Corpus assembly.** Shallow-cloned `TheSoftwareDesignLab/Kraken` (v2) and `TheSoftwareDesignLab/KrakenMobile` (v1); enumerated every `.feature` file. The master branches hold only one skeleton each; the real example corpora live on the **gh-pages branches**: 10 v1 examples (`examples/{askfm,f3,infinite-words,picap,quizup,shazam,spunky,stick,tictactoe,tumblr}`) and 3 v2 examples (`examples/{email,facebook,kahoot}`). These are exactly the papers' artifact sets: the Kraken 2.0 paper's links (`bit.ly/3oKPFcv`, `bit.ly/3Dz856j`) resolve to the project site backed by these branches; a scan of all 49 org repos plus repo search confirmed **no separate evaluation/replication repo exists**.
2. **Real-world extension.** GitHub-wide code search for the exact step text `"I send a signal to user"` in `.feature` files (58 hits). After removing skeleton copies and two verbatim copies of the canonical examples (`LensesResearchLab/ulos-executors`), this yielded **43 distinct real-world logical tests in 5 repos** — all Universidad de los Andes MISO/MISW-4103 course suites testing Ghost CMS on web with Kraken v2. Every file's per-actor action/signal sequence was extracted programmatically and inspected.
3. **Unit & classification rule.** Unit = one *logical test* (the set of `@userN` scenarios forming one choreography). **Causal ping-pong**: between every consecutive signal pair, exactly one actor performs app actions (others block waiting) — a total order expressed through signal plumbing. **Sustained simultaneity**: two or more actors perform app actions concurrently between sync points, where serializing would change what is tested. **Single-actor**: one actor, or N unsynchronized independent monologues (parallel-for-speed, serializable without changing what is tested). Concurrent *setup prologues* (both actors logging into their own accounts before the first signal) do not count as sustained simultaneity: serializing them changes wall-clock time only.

### A.2 Results

| Corpus | Logical tests | Causal ping-pong | Sustained simultaneity | Single-actor | Unclear |
|---|---|---|---|---|---|
| v1 canonical examples (KrakenMobile gh-pages) | 10 | 6 | **3** (QuizUp, Spunky, Stick) | 1 | 0 |
| v1 + v2 skeletons (`my_first.feature`) | 2 | 2 (degenerate handshakes) | 0 | 0 | 0 |
| v2 canonical examples (Kraken gh-pages) | 3 | 3 (email, facebook, **kahoot**) | 0 | 0 | 0 |
| Real-world v2 suites (5 repos, GitHub-wide) | 43 | 38 | 0 | 5 | 0 |
| **Total** | **58** | **49 (84.5%)** | **3 (5.2%)** | **6 (10.3%)** | **0** |

The 3 sustained-simultaneity cases, precisely: **QuizUp** (after the invitation handshake, both actors answer 7 quiz questions concurrently in a live duel), **Spunky** (both actors answer 14 trivia questions concurrently between `start`/`finished` barriers), **Stick Men Fight** (both actors run 1000-event monkeys concurrently in a shared multiplayer session, twice). All three are v1-era (2019) real-time-game demos authored by the tool's own authors.

### A.3 Findings

1. **Corpus-wide rate: 5.2% — below the ~10% threshold.** The v2-era slice, which is what the D6 hypothesis is actually about, is **0/47 (0%)**.
2. **The strongest single datum is v2's own Kahoot demo.** Kahoot is a real-time synchronized quiz — the ideal candidate for simultaneity — yet the v2 authors wrote it as pure ping-pong: the host performs zero steps while the player answers (`game_started` → player acts alone → `finished`). When even the flagship real-time demo serializes, v2's concurrency was the architecture, not the requirement.
3. **All 3 sustained cases fit the already-ratified detached-task escape hatch.** Each has the coarse *batch-overlap* shape: every actor runs one homogeneous batch of actions between two barriers. That is `starts <batch> in the background as "x"` + foreground batch + join — no fine-grained feature-file interleaving (lanes) is required by any scenario in the corpus. Strictly interpreted, **0% of the corpus needs lanes**.
4. **Real-world users invert the concern.** The 43 wild suites use signals overwhelmingly to *serialize*: one-way handoffs with progress-marker signal chains (up to 9 signals for one handoff), sequential pipelines of independent test cases chained by `done` signals (one with **20 actors**, 100% serialized), and 3 suites whose actors send signals nobody waits for (zero synchronization — racy parallel-for-speed). These users were hand-building v3's screenplay total order out of signal plumbing; several patterns (dangling signals, unsynchronized racing monologues) are exactly the failure modes the dry-run analyzer rejects statically.
5. **A recurring pattern worth naming: physical concurrency via persistent app state.** Shazam (music keeps playing after `press play` while the other actor recognizes it) and Kahoot (host screen stays live while the player answers) require the *devices* to be concurrently active but not the *actors* — a total order reproduces them exactly. This pattern needs no construct at all.

### A.4 Verdict

**D6 stands as ratified: screenplay total order remains the default; concurrent lanes remain deferred.** Corpus evidence: 5.2% sustained simultaneity overall (0% in the v2-era slice; 0% requiring lanes specifically), against the ~10% threshold of ADR-0001 §9.2.

**Tripwire (binding on Phase 2/3):** the lab-authored demo corpus alone scores 3/15 (20%) — real-time multiplayer games are exactly what the lab likes to demo. Therefore the examples suite must include at least one QuizUp/Spunky/Stick-shaped scenario (concurrent per-actor action batches between barriers) expressed with **detached tasks**, as a working proof that the escape hatch covers the 5%. If that proof fails or produces unacceptable ergonomics, this appendix's verdict is invalidated and the lanes decision must be reopened (the DAG scheduler keeps lanes additive — by design).

### A.5 Limitations (stated honestly)

- **The canonical corpus is examples-only** — 13 demos authored by the tool's authors to showcase capabilities, not user-authored regression suites. It may over-represent flashy simultaneity (games) relative to real testing needs; note that even so it lands at 20%, and the wild corpus lands at 0%.
- **The wild corpus is homogeneous**: one university course, one app under test (Ghost CMS), web-only, student-authored. It shows how Kraken v2 was actually used at scale, but not industrial multi-device practice.
- **Search bias**: GitHub code search covers public repos only and matched the exact English step text; private suites and non-signal-using suites are invisible. Because the query *required* a signal step, purely single-actor Kraken usage is under-counted — which biases the sample *toward* multi-user scenarios and therefore strengthens, not weakens, the ping-pong finding.
- **No independent-author corpus exists for the papers**: the SCP 2021/2023 and ICSE 2022 evaluations are usage demonstrations over the same example sets classified here; there is no user-study scenario dataset to check.
- Classification of the 3 sustained cases as detached-task-expressible is an analytical claim, not yet an executed proof — hence the A.4 tripwire.

## Appendix B — VS Code autocomplete spike findings (mandated by ADR-0001 §5.9/§7)

**Status:** Complete (2026-07-03), confidence: high. Source-code analysis of @cucumber/language-service (the engine behind the official CucumberOpen.cucumber-official extension).

### B.1 How discovery actually works

The CucumberOpen.cucumber-official extension launches @cucumber/language-server, which globs workspace files matching the cucumber.glue setting (src/fs.ts: loadGlueSources), maps file extension to a tree-sitter grammar via glueLanguageNameByExt ('.ts'/'.cts'/'.mts'/'.tsx' -> 'tsx' grammar; '.js'/'.cjs'/'.mjs'/'.jsx' -> 'javascript'), parses each file with web-tree-sitter, and runs @cucumber/language-service's tsxLanguage.defineStepDefinitionQueries. The TS query is: (call_expression function: (identifier) @function-name arguments: (arguments [(string)(regex)(template_string)] @expression) (#match? @function-name "Given|When|Then")). Discovery is PURELY LEXICAL: a call whose callee is a BARE IDENTIFIER matching the unanchored regex Given|When|Then, whose argument list contains a LITERAL string, regex literal, or template string WITHOUT ${} substitutions. Import sources are never examined (any npm package qualifies); renamed imports match only if the new name still contains Given/When/Then as a substring; member/namespace calls like steps.Given(...) or k.When(...) do NOT match in .ts files (only the compiled-CJS forms (0, pkg_1.Given)(...) and (0, Given)(...) match, and only in .js files via javascriptLanguage's extra queries); the step callback is never inspected, so Kraken's context-first (ctx, ...) handler signature is invisible to the extension. Matched expression strings are then parsed by @cucumber/cucumber-expressions against a ParameterTypeRegistry assembled in ExpressionBuilder.build (settings-provided parameterTypes registered first, then source-discovered defineParameterType calls) to drive completion, undefined-step diagnostics, and go-to-definition in .feature files.

### B.2 What this fixed in @kraken/gherkin's API

Export top-level named functions literally called Given, When, Then (context-first, typed, cucumber-expressions-matched) plus defineParameterType({name, regexp, transformer}) from @kraken/gherkin — i.e. option (a) from ADR-0001 §5.9. This is a superset of what the queries match and needs zero extra tooling: import { Given, When, Then, defineParameterType } from '@kraken/gherkin' just works because discovery is name-based and import-source-blind. If the steps() factory variant is kept for scoping, document it as MUST-destructure: const { Given, When, Then } = steps(...) works; namespaced s.Given(...) silently loses autocomplete — safer to make the factory an additive convenience, not the primary API. A generated shim is unnecessary. Keep defineParameterType's options-object shape with literal name and regexp at call sites (regex literal preferred) so user-defined types index; avoid builder-style or computed registration in documented examples. For Kraken's BUILT-IN {actor} type and built-in steps (which live in node_modules, outside glue globs): have kraken init scaffold .vscode/settings.json with cucumber.parameterTypes for {actor}, and optionally add node_modules/@kraken/gherkin/dist/**/*.js to cucumber.glue so built-in steps index via the compiled-CJS queries (manual-verify item). Also consider shipping a cucumber.snippetTemplates override so generated snippets include the ctx parameter.

Implemented accordingly: `createStepRegistry()` returns destructurable `Given`/`When`/`Then`/`defineParameterType` — call sites are bare identifiers with literal expressions, which the tree-sitter queries match regardless of import source. Documented rule: **always destructure** (namespaced `api.Given(...)` silently loses autocomplete in .ts files).

### B.3 Editor settings a Kraken project needs

Because the built-in {actor}/{duration} parameter types live inside node_modules (outside the extension's glue globs), `kraken init` (Phase 2) must scaffold:

```jsonc
{
  "cucumber.features": ["features/**/*.feature"],
  "cucumber.glue": ["features/**/*.ts", "features/**/*.tsx"],
  "cucumber.parameterTypes": [
    { "name": "actor", "regexp": "[a-zA-Z][a-zA-Z0-9_-]*|\\"[^\\"]+\\"" }
  ]
}
// Notes: (1) the glue/features values above match the extension defaults for TS projects — only override cucumber.glue if Kraken scaffolds steps elsewhere (e.g. "kraken/steps/**/*.ts"); (2) cucumber.parameterTypes entries take name + regexp-as-string only (compiled with new RegExp) and are needed solely for types NOT defined via a defineParameterType call inside the glue globs — Kraken's built-in {actor} qualifies; keep the regexp byte-identical to the runtime one; (3) optional, to index built-in steps shipped in the package: add "node_modules/@kraken/gherkin/dist/**/*.js" to cucumber.glue (compiled (0, x.Given) calls are matched in .js files); (4) optional, to make generated snippets context-first: "cucumber.snippetTemplates": { "tsx": "{{ keyword }}('{{ expression }}', async (ctx{{ #parameters }}, {{ name }}: {{ type }}{{ /parameters }}) => {\n  // {{ blurb }}\n})" }
```

The regexps must stay byte-identical to `ACTOR_REGEXP`/`DURATION_REGEXP` exported by @kraken/gherkin.

### B.4 Residual 10-minute manual checklist (only a human with VS Code can verify)

- Install CucumberOpen.cucumber-official in a real VS Code, open the scaffolded Kraken example project, and wait for indexing to finish (watch the Cucumber output channel for errors).
- In a .feature file, type 'When ' and confirm completion offers the Kraken step with {actor}/{string} rendered as tab-stop placeholders; accept it and confirm no 'undefined step' underline.
- F12 / cmd-click a matched step line and confirm Go to Definition lands on the TypeScript Given/When/Then call in the step file.
- Edit a step expression in the .ts file, save, and confirm the .feature diagnostics/completions update without reloading the window (reindex-on-save behavior).
- Prove the settings-based {actor} is load-bearing: temporarily delete the cucumber.parameterTypes entry, reload, confirm {actor} steps go undefined; restore it.
- Alternative path: add defineParameterType({ name: 'actor', regexp: /[A-Z][a-zA-Z0-9_]*/, transformer: ... }) in a glue-globbed support file (with the setting removed) and confirm equivalent autocomplete — decide which mechanism kraken init scaffolds.
- Built-in steps: add node_modules/@kraken/gherkin/dist/**/*.js to cucumber.glue and verify the compiled (0, x.Given)(...) calls are indexed and go-to-definition lands somewhere sane; note indexing latency impact.
- If shipping the steps() factory variant, write one file using const { Given, When, Then } = steps(...) and confirm those steps index too.
- Trigger snippet generation for an undefined step and check the default tsx template output; if the ctx-first cucumber.snippetTemplates override is scaffolded, confirm it renders correctly.
- Scan the Cucumber output channel for tree-sitter parse errors on real Kraken step files — .ts is parsed with the tsx grammar, so old-style angle-bracket casts (<T>expr) can break parsing of a whole file; confirm Kraken's emitted/scaffolded code avoids them.
