## Part A ‚Äî The event schema (as built in Phase 1, documented here)

*This part was implemented with Phase 1 (the schema is the spine everything subscribes to) and is recorded as-built; writing it down was deferred to this document.*

### A1 ‚Äî One flat discriminated union

`KrakenEvent` (in `@kraken/contracts`) is a flat union discriminated on `type`, 16 members across five families:

- **run**: `runStarted`, `runFinished`
- **scenario**: `scenarioStarted` (carries the `ActorSummary[]` cast), `scenarioFinished`
- **step**: `stepStarted`, `stepFinished` (status: passed | failed | skipped, durationMs)
- **actors/drivers**: `actorSessionStarted`, `actorSessionFinished`, `driverRegistered`, `driverDisabled`, `driverLog`
- **signals** (the differentiator gets first-class events): `signalSent`, `signalReceived` (with `latencyMs`), `signalWaitStarted`, `signalTimedOut`
- **artifacts**: `artifactCaptured` (kind: screenshot | source | log | video)

Every event carries `{ ts, runId, seq }` (`seq` assigned by the core EventBus ‚Äî total order within a run). Reporters receive events through the `Reporter` contract (`{ id, onEvent }`); reporter exceptions are isolated by the bus and surfaced via `onReporterError` ‚Äî a broken reporter can never fail a run.

### A2 ‚Äî Schema stability contract

The zod schemas for every member live module-private in `@kraken/core` (zod never leaks into the public surface); a **JSON-Schema snapshot** (`kraken-events.schema.json`) is regenerated by test and any drift fails `pnpm check`. Evolution rules: adding a member or an optional field is minor; renaming/removing/retyping is breaking and requires a contract major bump (same regime as the SPI ‚Äî exercised for real at contract 2.0).

### A3 ‚Äî Reporters are projections

The JSONL event log (one event per line, written per run under `.kraken/runs/<runId>/events.jsonl`) is the substrate; every other reporter is a pure projection of the same stream ‚Äî the property the TUI (ADR-0005 D3), the M1 demos' analysis, and the future `kraken serve` GUI all rely on. No reporter may reach into the runner.

## Part B ‚Äî The Phase 4 reporter set (Allure 3 + CTRF), as verified and built

### B1 ‚Äî Allure 3 (allure-js-commons `/sdk/reporter`)

The custom-integration path live-verified 2026-07-05: `ReporterRuntime` + `FileSystemWriter` (allure-js-commons ~3.10.2). **HTML generation is pure Node** ‚Äî the `allure` npm CLI (3.14.x, from the Allure 3 TypeScript rewrite) has zero jars; students never need Java. Multi-actor model (verified to render): one Allure test per scenario, actor‚Üîplatform cast as test parameters, one top-level step per Kraken step named `[actor] ‚Ä¶` plus a machine-readable `actor` step parameter, and **signal waits/receptions/timeouts as first-class steps** ‚Äî the choreography reads directly in the report. `writeTest` fires at `scenarioFinished` (crash-safe); all timestamps propagate from Kraken events. Caveat encoded in code: `startStep` returns `string | undefined` (the runtime console-errors instead of throwing on misuse).

### B2 ‚Äî CTRF (hand-emitted, zero runtime deps)

CTRF is a pre-1.0 Working Draft; `specVersion` is pinned to `"0.0.0"` (what the reference implementation emits ‚Äî upstream examples inconsistently say 1.0.0; both pass the schema). The root requires `reportFormat: 'CTRF'`; `additionalProperties: false` everywhere means Kraken extras ride in `extra`. The **official JSON Schema is vendored** into the reporters package and an ajv test validates every emitted shape ‚Äî drift in either direction fails `pnpm check`. The `ctrf` npm helper was evaluated and rejected as a runtime dep (4 transitive deps, 0.x churn, lagging schema); its CLI namesake gotcha (a deprecated local bin shadowing `ctrf-cli` via npx and silently exiting 0) is recorded here as a warning for CI wiring later.

### B3 ‚Äî Wiring and the cucumber-messages question

Both reporters are always-on in `kraken run` (projections are cheap; the run dir gains `allure-results/` + `ctrf-report.json` next to `events.jsonl`). A Cucumber-messages compatibility stream was considered (ADR-0001 ¬ß5.12) and **deferred with evidence**: Allure's own HTML replaced the strongest consumer (report UIs), CTRF covers CI summaries, and no concrete consumer for ndjson messages surfaced in Phases 1-4. Revisit only when a real formatter need lands.

## Consequences

- Anything expressible as an event-stream projection needs zero core changes; anything NOT expressible that way is a design smell caught at review.
- The JSONL log makes every run's full choreography (including signal latencies) inspectable with `grep`/`jq` ‚Äî used repeatedly during the M1/Phase-3 device debugging.
